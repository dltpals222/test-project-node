# 아키텍처 개요

전체 구조와 프론트엔드↔백엔드가 어떻게 맞물리는지 설명합니다.

## 모노레포 구조

하나의 저장소에서 프론트엔드와 백엔드를 함께 관리합니다.

```
test-project-node/
├── frontend/        # Next.js (App Router) — 사용자 화면
├── backend/         # NestJS — REST API 서버
├── docs/            # 프로젝트 문서
└── CLAUDE.md        # 문서 허브
```

- 각 앱은 **독립적인 `package.json`** 을 가집니다 (별도 의존성, 별도 실행).
- 두 앱은 서로의 소스를 직접 import 하지 않습니다. 통신은 **HTTP(REST)** 로만 합니다.

## 통신 방식

- 프론트엔드는 백엔드의 REST API를 호출해 데이터를 주고받습니다.
- API 기본 경로는 `/api` 프리픽스를 사용합니다. (예: `GET /api/users`)
- 요청/응답은 JSON을 사용합니다.

```
[브라우저]
   │  (사용자 요청)
   ▼
[Next.js / frontend]  ──HTTP(JSON)──▶  [NestJS / backend]  ──▶  [DB]
   │                                          │
   └────────◀── 응답(JSON) ───────────────────┘
```

- **데이터 조회**는 가능하면 서버 컴포넌트에서 백엔드를 직접 호출합니다. (자세한 패칭 규칙은 [프론트엔드 컨벤션](frontend-conventions.md) 참고)
- 민감한 키가 필요한 호출은 절대 브라우저에서 직접 하지 않고, 서버 측(서버 컴포넌트 / Route Handler)을 거칩니다.

## 환경변수 분리

각 앱은 자기 디렉토리의 `.env` 파일을 사용합니다. `.env` 는 커밋하지 않고, `.env.example` 만 커밋합니다.

| 위치 | 예시 키 | 비고 |
|------|---------|------|
| `frontend/.env.local` | `NEXT_PUBLIC_API_URL` | 백엔드 주소. `NEXT_PUBLIC_` 접두사가 붙은 값만 브라우저에 노출됨 |
| `backend/.env` | `PORT`, `DATABASE_URL` | 서버 전용. 절대 브라우저에 노출하지 않음 |

- **규칙:** 비밀 값(DB 비밀번호, API 시크릿)은 백엔드에만 둡니다. 프론트엔드에는 `NEXT_PUBLIC_` 으로 노출해도 안전한 값만 둡니다.

## 책임 경계

- **frontend:** UI 렌더링, 라우팅, 사용자 입력 처리, 백엔드 API 호출.
- **backend:** 비즈니스 로직, 데이터 검증, DB 접근, 인증/인가.
- 비즈니스 규칙(권한 체크, 금액 계산 등)은 **백엔드를 단일 진실 공급원**으로 둡니다. 프론트엔드의 검증은 UX 보조용이며, 보안 경계가 아닙니다.
