# 02. 보안 구현 명세

원본 프로젝트의 보안 메커니즘을 **동일 수준**으로 재현하기 위한 명세입니다.
모든 코드는 원본 패턴 기반이며, 도메인 이름만 더미 기준(`role` 4단계 등)으로 치환했습니다.

---

## 1. 인증 (Authentication) — JWT

### 1-1. 방식 요약

- **Access Token 단독** 구조 (Refresh Token 없음 — 원본과 동일하게 생략)
- 만료: **7일** (`expiresIn: '7d'`)
- 페이로드: `{ sub: userId, username, role }`
- 라이브러리: `jsonwebtoken`(직접 sign), `@nestjs/passport` + `passport-jwt`(검증)

### 1-2. 토큰 발급 (`auth.service.ts`)

```typescript
import * as jwt from 'jsonwebtoken';

const payload = { sub: user.id, username: user.username, role: user.role };

const accessToken = jwt.sign(
  payload,
  process.env.JWT_SECRET || 'dev-secret-key', // ⚠️ 아래 "보안 주의" 참조
  { expiresIn: '7d' },
);
```

### 1-3. 토큰 검증 (`common/guards/jwt.strategy.ts`)

```typescript
import { ExtractJwt, Strategy } from 'passport-jwt';
import { PassportStrategy } from '@nestjs/passport';

@Injectable()
export class JwtStrategy extends PassportStrategy(Strategy) {
  constructor() {
    super({
      jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
      ignoreExpiration: false,
      secretOrKey: process.env.JWT_SECRET || 'dev-secret-key',
    });
  }

  async validate(payload: any) {
    // request.user 로 주입됨
    return { id: payload.sub, username: payload.username, role: payload.role };
  }
}
```

> ⚠️ **보안 주의 (원본 약점 — 더미에서 고칠 것)**: 원본은 `JWT_SECRET` 환경변수가 정의되어 있지 않아 `'dev-secret-key'` 하드코딩 폴백이 항상 사용됩니다. 더미에서는 **`.env`에 `JWT_SECRET`를 반드시 정의**하고, 가능하면 폴백을 제거해 미정의 시 부팅 실패하도록 만드세요. (동일 "수준"을 넘어 한 단계 개선)

## 2. 인가 (Authorization) — RBAC

### 2-1. 역할 모델 (더미 치환)

```typescript
// users/entities/user.entity.ts
export type UserRole = 'superadmin' | 'admin' | 'manager' | 'member';
export type AccountStatus = 'active' | 'suspended';
```

| 역할 | 권한 개념 |
| --- | --- |
| `superadmin` | 시스템 전체 관리, 다른 admin 비번 초기화 가능 |
| `admin` | 일반 관리. **다른 admin 계정의 비번은 초기화 불가** (superadmin만) |
| `manager` | 내부 담당자. 배정된 레코드만 상태 변경·재배정 |
| `member` | 외부 당사자. 배정된 레코드만 상태 변경 |

### 2-2. `@Roles` 데코레이터 (`common/decorators/roles.decorator.ts`)

```typescript
import { SetMetadata } from '@nestjs/common';

export const ROLES_KEY = 'roles';
export const Roles = (...roles: string[]) => SetMetadata(ROLES_KEY, roles);
```

### 2-3. `RolesGuard` (`common/guards/roles.guard.ts`)

```typescript
@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    const requiredRoles = this.reflector.getAllAndOverride<string[]>(ROLES_KEY, [
      context.getHandler(),
      context.getClass(),
    ]);

    if (!requiredRoles) return true; // 역할 미지정 → 통과

    const { user } = context.switchToHttp().getRequest();
    if (!requiredRoles.includes(user.role)) {
      throw new ForbiddenException('Insufficient permissions');
    }
    return true;
  }
}
```

### 2-4. `JwtAuthGuard` (`common/guards/jwt-auth.guard.ts`)

```typescript
import { AuthGuard } from '@nestjs/passport';

@Injectable()
export class JwtAuthGuard extends AuthGuard('jwt') {}
```

### 2-5. 컨트롤러 적용 패턴

```typescript
@Controller('records')
export class RecordsController {
  @Patch(':id/status')
  @UseGuards(JwtAuthGuard, RolesGuard)   // 인증 → 인가 순서
  @Roles('superadmin', 'admin', 'manager')
  updateStatus(@Param('id') id: string, @Body() dto: UpdateStatusDto) { /* ... */ }
}
```

> **글로벌 가드 없음**: 원본은 전역 가드를 두지 않고, 보호가 필요한 엔드포인트마다 `@UseGuards`를 명시합니다. 가드 미부착 엔드포인트는 곧 공개 엔드포인트입니다(별도 `@Public` 데코레이터 없음). 더미도 동일하게 명시적 부착 방식을 따르세요.

### 2-6. 비즈니스 규칙 인가 예시 (서비스 레이어)

```typescript
// admin은 다른 admin의 비밀번호를 초기화할 수 없음 (superadmin만 가능)
if (actor.role === 'admin' && target.role === 'admin') {
  throw new ForbiddenException('관리자는 다른 관리자 계정의 비밀번호를 초기화할 수 없습니다');
}
```

> RBAC는 Guard(엔드포인트 레벨) + 서비스 레이어(데이터 소유권/계층 규칙)의 2단으로 구성합니다. 배정 기반 접근(배정된 레코드만)은 `record_assignments` 조인으로 서비스에서 강제합니다(03 문서).

## 3. 비밀번호 보안

### 3-1. 해싱 — bcrypt (salt round 10)

```typescript
import * as bcrypt from 'bcrypt';

// 저장 시
const passwordHash = await bcrypt.hash(plainPassword, 10);

// 검증 시
const isValid = await bcrypt.compare(plainPassword, user.password_hash);
```

- 저장 컬럼: `password_hash` (평문 저장 절대 금지)

### 3-2. 비밀번호 정책 (DTO 검증)

```typescript
// auth/dto/login.dto.ts (또는 change-password.dto.ts)
@MinLength(8)
@Matches(
  /^(?=.*[a-zA-Z])(?=.*\d)(?=.*[!@#$%^&*()_+\-=\[\]{};':"\\|,.<>\/?~`])/,
  { message: '비밀번호는 영문, 숫자, 특수문자를 포함하여 8자 이상이어야 합니다' },
)
password: string;
```

### 3-3. 임시 비밀번호 생성 (`common/utils/password.utils.ts`)

```typescript
// 12자, 소문자/대문자/숫자/특수문자 각 1개 이상 보장 후 셔플
export function generateTempPassword(length = 12): string {
  const lower = 'abcdefghijklmnopqrstuvwxyz';
  const upper = 'ABCDEFGHIJKLMNOPQRSTUVWXYZ';
  const digit = '0123456789';
  const special = '!@#$%^&*';
  // 각 그룹에서 최소 1개 + 나머지 랜덤 → 셔플
  // ...
}
```

### 3-4. 비밀번호 확인 일치 검증 (커스텀 데코레이터)

```typescript
// change-password.dto.ts
@IsEqualTo('newPassword', { message: '새 비밀번호와 확인이 일치하지 않습니다' })
confirmPassword: string;
```

`@IsEqualTo`는 `class-validator`의 `registerDecorator`로 직접 구현합니다 (`common/decorators/`).

## 4. 로그인 보안 (계정 잠금 + 강제 변경)

원본의 핵심 방어 로직입니다. 더미에서 **그대로 재현**하세요.

### 4-1. 5회 실패 → 15분 잠금

```typescript
const MAX_LOGIN_ATTEMPTS = 5;
const LOCK_DURATION_MINUTES = 15;

// 1) 잠금 상태 확인
if (user.locked_until && new Date(user.locked_until) > new Date()) {
  throw new UnauthorizedException('계정이 일시적으로 잠겼습니다. 잠시 후 다시 시도하세요');
}

// 2) 비번 불일치 시 카운트 증가
if (!isValid) {
  const attempts = user.failed_login_attempts + 1;
  if (attempts >= MAX_LOGIN_ATTEMPTS) {
    const lockedUntil = new Date(Date.now() + LOCK_DURATION_MINUTES * 60 * 1000);
    await this.usersRepository.lockAccount(user.id, lockedUntil);
  } else {
    await this.usersRepository.incrementFailedAttempts(user.id);
  }
  throw new UnauthorizedException('Invalid credentials');
}

// 3) 성공 시 카운트 리셋
await this.usersRepository.resetFailedAttempts(user.id);
```

관련 컬럼: `failed_login_attempts INT DEFAULT 0`, `locked_until TIMESTAMP NULL` (03 문서).

### 4-2. 계정 상태 확인

```typescript
if (user.status !== 'active') {
  throw new UnauthorizedException('Account is suspended');
}
```

### 4-3. 첫 로그인 강제 비밀번호 변경

```typescript
// password_changed_at 이 NULL 이면 = 임시 비번 상태 = 변경 강제
const requiresPasswordChange = !user.password_changed_at;
if (!requiresPasswordChange) {
  await this.usersRepository.updateLastLogin(user.id);
}
// 로그인 응답에 requiresPasswordChange 플래그 포함 → 프론트가 비번 변경 화면으로 유도
```

## 5. 요청 검증 (ValidationPipe)

### 5-1. 전역 등록 (`app.module.ts`)

```typescript
import { APP_PIPE } from '@nestjs/core';
import { ValidationPipe } from '@nestjs/common';

@Module({
  providers: [
    {
      provide: APP_PIPE,
      useValue: new ValidationPipe({
        whitelist: true,            // DTO 미정의 필드 자동 제거
        forbidNonWhitelisted: false, // 원본 설정 (true 권장 — 아래 주의 참조)
        transform: true,            // 타입 자동 변환
      }),
    },
  ],
})
export class AppModule {}
```

> ⚠️ 원본은 `forbidNonWhitelisted: false`라 미정의 필드가 와도 거부하지 않고 제거만 합니다. 동일 수준 유지 시 그대로, 한 단계 강화 시 `true`로 설정해 미정의 필드 포함 요청을 차단하세요.

### 5-2. DTO 검증 패턴 (class-validator)

```typescript
export class CreateRecordDto {
  @IsString()
  @MaxLength(150)
  name: string;

  @IsEnum(RecordStatus)
  status: RecordStatus;

  @IsBoolean()
  @IsOptional()
  consent?: boolean;

  @ValidateIf((o) => o.type === 'special')
  @IsString()
  extraField?: string;
}
```

## 9. 프론트엔드 보안

### 9-1. axios 토큰 처리 (`lib/api.ts`)

```typescript
const STORAGE_KEY = 'dummy_access_token';

const api = axios.create({ baseURL: '/api', timeout: 5000 });

// 요청 인터셉터: 토큰 자동 첨부
api.interceptors.request.use((config) => {
  const token = localStorage.getItem(STORAGE_KEY);
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// 응답 인터셉터: 401 → 토큰 폐기
api.interceptors.response.use(
  (res) => res,
  (error) => {
    if (error.response?.status === 401) {
      localStorage.removeItem(STORAGE_KEY);
      // 쿠키도 제거 (middleware용)
    }
    return Promise.reject(error);
  },
);
```

- 토큰 저장: `localStorage` + `Cookie`(middleware가 읽음, 7일)

### 9-2. Next.js middleware 인증 (`middleware.ts`)

```typescript
import { jwtVerify } from 'jose';   // Edge 런타임 JWT 파싱

export async function middleware(req: NextRequest) {
  const PUBLIC_PATHS = ['/login', '/'];
  const path = req.nextUrl.pathname;
  if (PUBLIC_PATHS.includes(path)) return NextResponse.next();

  const token = req.cookies.get('dummy_access_token')?.value;
  if (!token) return NextResponse.redirect(new URL('/login', req.url));

  // 역할별 기본 페이지 리다이렉트 / 비번 변경 필요 상태 처리 / 역할 제한 경로 차단
  // 예: /admin/* → admin·superadmin 만, 그 외 → 자기 역할 홈으로
}
```

| 역할 | 기본 진입 경로(예) |
| --- | --- |
| `superadmin` / `admin` | `/admin` |
| `manager` | `/manager` |
| `member` | `/member` |

> 개발 편의를 위해 `npm run dev`는 인증 우회, `npm run dev:auth`(`ENABLE_AUTH=true`)는 인증 활성화로 분리합니다.

> ⚠️ **프론트 검증은 UX 보조일 뿐**입니다. 실제 접근 통제는 항상 백엔드 Guard + 서비스 레이어에서 강제하세요. middleware만 믿으면 안 됩니다.
