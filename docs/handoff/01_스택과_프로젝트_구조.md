# 01. 스택과 프로젝트 구조

원본 프로젝트와 동일한 프레임워크 버전·구조로 더미를 셋업하기 위한 명세입니다.
버전 문자열은 원본 `package.json` 기준이며, 더미에서도 **동일 메이저 버전**을 사용하세요.

### 1-3. 디렉토리 구조

```
backend/
├── package.json
├── tsconfig.json          # target ES2023, module commonjs, decorators 활성화
├── knexfile.js            # 환경별 DB 설정 (03 문서 참조)
├── .env.example           # 환경변수 템플릿
├── migrations/            # Knex 마이그레이션 (03 문서 참조)
├── seeds/                 # 시드 (superadmin 계정 등)
└── src/
    ├── main.ts            # 부트스트랩 (dotenv, CORS, Swagger, port 3001)
    ├── app.module.ts      # 모듈 등록 + 전역 ValidationPipe + KnexModule
    ├── common/
    │   ├── decorators/    # @Roles, @IsEqualTo 등 커스텀 데코레이터
    │   ├── guards/        # jwt-auth.guard.ts, roles.guard.ts, jwt.strategy.ts
    │   └── utils/         # password.utils.ts (임시 비번 생성 등)
    ├── config/            # (더미에서는 최소화)
    ├── database/          # advisory-locks 등 동시성 유틸 (선택)
    └── modules/           # 도메인 모듈 (아래)
        ├── auth/          # 로그인, 비번 변경, 비번 초기화
        ├── users/         # 사용자 CRUD + entity
        ├── partners/      # 외부/내부 당사자 (원본 parties)
        ├── records/       # 핵심 레코드 (원본 leads)
        ├── assignments/   # 레코드 배정 (원본 lead-assignments)
        └── identity/      # 본인인증 더미 (원본 identity)
```

> **모듈 패턴**: 각 모듈은 `dto/`, `entities/`, `*.controller.ts`, `*.service.ts`, `*.repository.ts`, `*.module.ts` 구성. Repository 패턴으로 Knex 쿼리를 캡슐화합니다.

### 1-4. `main.ts` 부트스트랩 골격

```typescript
import * as dotenv from 'dotenv';
import * as path from 'path';

// 1) NODE_ENV 기반 env 파일 로드 (local → .env.local, 그 외 → .env.{env})
const env = process.env.NODE_ENV || 'local';
const envFile = env === 'local' ? '.env.local' : `.env.${env}`;
dotenv.config({ path: path.resolve(process.cwd(), envFile) });

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  // 2) CORS — origin 화이트리스트 (02 문서 참조)
  app.enableCors({ /* origin 배열, credentials: true */ });

  // 3) Swagger — 프로덕션 제외
  if (process.env.NODE_ENV !== 'production') {
    // SwaggerModule.setup('api-docs', app, document)
  }

  // 4) 전역 prefix /api (프론트 프록시와 정합)
  app.setGlobalPrefix('api');

  await app.listen(process.env.PORT || 3001, '0.0.0.0');
}
```

> 전역 `ValidationPipe`는 `main.ts`가 아니라 `app.module.ts`에서 `APP_PIPE`로 등록합니다 (02 문서 참조).

### 2-4. API 프록시 (`next.config.ts`)

```typescript
const nextConfig = {
  async rewrites() {
    return [
      {
        source: "/api/:path*",
        destination: `${process.env.API_URL || "http://localhost:3001"}/api/:path*`,
      },
    ];
  },
};
```

> 프론트는 항상 상대경로 `/api/*`로 호출하고, Next가 백엔드로 프록시합니다. 환경별 `API_URL`만 바꾸면 됩니다.

## 3. 인프라 (Docker)

### `docker-compose.yml` (로컬 PostgreSQL)

```yaml
services:
  postgres-dev:
    image: postgres:15-alpine
    container_name: dummy-postgres-dev
    environment:
      POSTGRES_USER: dummy
      POSTGRES_PASSWORD: ${LOCAL_DB_PASSWORD:-dev_password_123}
      POSTGRES_DB: dummy_dev
    ports:
      - "5432:5432"
    volumes:
      - postgres-dev-data:/var/lib/postgresql/data

volumes:
  postgres-dev-data:
```

## 4. 셋업 순서

```bash
# 1) DB 컨테이너 기동
docker compose up -d postgres-dev

# 2) 백엔드
cd backend
npm install
cp .env.example .env.local   # 값 채우기 (JWT_SECRET 반드시 정의!)
npm run db:migrate:local
npm run db:seed:local        # superadmin 시드
npm run dev                  # http://localhost:3001/api

# 3) 프론트엔드
cd ../frontend
npm install
npm run dev                  # http://localhost:3000
```
