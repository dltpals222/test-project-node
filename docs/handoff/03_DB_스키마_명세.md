# 03. DB 스키마 명세 (핵심 + 보안 테이블)

- **DB**: PostgreSQL 15 (원본은 11+ 호환, 더미는 15 사용)
- **ORM**: Knex.js 3 (`knex` ^3.1.0, `pg` ^8.19.0, `nest-knexjs` ^0.0.34)
- **범위**: 인증·권한·핵심 워크플로에 필요한 **8개 테이블 + 4개 ENUM**만 포함.
  원본의 CMS/상담시간/광고추적/부정신청 도메인 테이블은 **제외**.
- **도메인**: 범용 치환 (README 치환표 참조). **보안 컬럼명은 원본 그대로 보존**.

---

## 1. Knex 설정 (`knexfile.js`)

```javascript
require('dotenv').config({
  path: process.env.NODE_ENV === 'local' || !process.env.NODE_ENV
    ? '.env.local'
    : `.env.${process.env.NODE_ENV}`,
});

const base = {
  client: 'pg',
  migrations: { tableName: 'knex_migrations', directory: './migrations' },
  seeds: { directory: './seeds' },
};

module.exports = {
  local: {
    ...base,
    connection: {
      host: process.env.DB_HOST || 'localhost',
      port: Number(process.env.DB_PORT) || 5432,
      user: process.env.DB_USER || 'dummy',
      password: process.env.DB_PASSWORD || 'dev_password_123',
      database: process.env.DB_NAME || 'dummy_dev',
    },
  },
  development: {
    ...base,
    connection: {
      host: process.env.DB_HOST,
      port: Number(process.env.DB_PORT),
      user: process.env.DB_USER,
      password: process.env.DB_PASSWORD,
      database: process.env.DB_NAME,
      ssl: process.env.DB_SSL === 'true' ? { rejectUnauthorized: false } : false,
    },
  },
  production: { /* development 과 동일 + ssl 강제 */ },
};
```

- **타임존**: KnexModule 연결 시 `+09:00` 등 고정(원본 패턴). 더미는 필요에 맞게.
- **ID 생성**: 모든 테이블 `BIGSERIAL` (auto-increment 64bit)
- **타임스탬프 컨벤션**: `created_at` / `updated_at` / `deleted_at`, 타입 `TIMESTAMP` 또는 `TIMESTAMPTZ`, 기본값 `knex.fn.now()`

## 2. ENUM 타입

```sql
-- role (원본 user_role 치환: 4단계 RBAC)
CREATE TYPE role AS ENUM ('superadmin', 'admin', 'manager', 'member');

-- account_status (그대로)
CREATE TYPE account_status AS ENUM ('active', 'suspended');

-- record_status (원본 lead_status 치환: 범용 워크플로)
CREATE TYPE record_status AS ENUM ('new', 'assigned', 'in_progress', 'completed', 'cancelled');

-- assignment_status (그대로)
CREATE TYPE assignment_status AS ENUM ('assigned', 'removed');
```

## 3. 테이블 명세

### 3-1. `users` — 계정/인증 핵심 ⭐보안

| 컬럼 | 타입 | 제약 | 비고 |
| --- | --- | --- | --- |
| `id` | BIGSERIAL | PK | |
| `username` | VARCHAR | UNIQUE, NOT NULL | 로그인 ID |
| `password_hash` | VARCHAR | NOT NULL | bcrypt 해시 |
| `role` | role(ENUM) | NOT NULL | RBAC |
| `status` | account_status | DEFAULT 'active' | 정지 시 로그인 차단 |
| `failed_login_attempts` | INTEGER | DEFAULT 0 | ⭐로그인 잠금 |
| `locked_until` | TIMESTAMP | NULL | ⭐잠금 해제 시각 |
| `password_changed_at` | TIMESTAMP | NULL | ⭐NULL = 첫 로그인 강제변경 |
| `suspended_at` | TIMESTAMP | NULL | 정지 시각 |
| `last_login_at` | TIMESTAMP | NULL | |
| `created_at` | TIMESTAMP | DEFAULT now() | |
| `updated_at` | TIMESTAMP | DEFAULT now() | |

> ⭐ 표시 4개 컬럼은 보안 메커니즘(02 문서 §4)에 직결되므로 반드시 포함하세요.

### 3-2. `partners` — 외부/내부 당사자 (원본 parties)

| 컬럼 | 타입 | 제약 | 비고 |
| --- | --- | --- | --- |
| `id` | BIGSERIAL | PK | |
| `user_id` | BIGINT | UNIQUE, FK→users(id) ON DELETE CASCADE | 1:1 |
| `partner_type` | VARCHAR(20) | NOT NULL | 'external' \| 'internal' |
| `name` | VARCHAR(100) | NOT NULL | |
| `status` | VARCHAR(20) | DEFAULT 'active' | |
| `created_at` | TIMESTAMP | DEFAULT now() | |
| `updated_at` | TIMESTAMP | DEFAULT now() | |

인덱스: `partner_type`, `status`, `user_id`

### 3-3. `records` — 핵심 워크플로 레코드 (원본 leads) ⭐soft delete

| 컬럼 | 타입 | 제약 | 비고 |
| --- | --- | --- | --- |
| `id` | BIGSERIAL | PK | |
| `name` | VARCHAR | NOT NULL | |
| `phone` | VARCHAR | NOT NULL | |
| `status` | record_status | DEFAULT 'new' | |
| `identity_verification_id` | BIGINT | FK→identity_verifications(id) ON DELETE SET NULL | |
| `assigned_at` | TIMESTAMP | NULL | |
| `completed_at` | TIMESTAMP | NULL | |
| `deleted_at` | TIMESTAMPTZ | NULL | ⭐soft delete |
| `deleted_by` | BIGINT | FK→users(id) ON DELETE SET NULL | ⭐삭제 감사 |
| `restored_at` | TIMESTAMPTZ | NULL | ⭐복원 |
| `restored_by` | BIGINT | FK→users(id) ON DELETE SET NULL | ⭐복원 감사 |
| `created_at` | TIMESTAMP | DEFAULT now() | |
| `updated_at` | TIMESTAMP | DEFAULT now() | |

인덱스:
- `phone`, `created_at`, `status`
- `idx_records_active`: `(id) WHERE deleted_at IS NULL` (활성 조회 최적화)
- `idx_records_archive`: `(deleted_at DESC) WHERE deleted_at IS NOT NULL` (보관함)

> ⭐ soft delete 4종 컬럼은 "누가 언제 삭제/복원했는가"를 남기는 감사 패턴입니다. 보안·추적 측면에서 그대로 보존하세요.

### 3-4. `record_assignments` — 레코드 배정 (원본 lead_assignments) ⭐RBAC 범위

| 컬럼 | 타입 | 제약 | 비고 |
| --- | --- | --- | --- |
| `id` | BIGSERIAL | PK | |
| `record_id` | BIGINT | FK→records(id) ON DELETE CASCADE | |
| `partner_id` | BIGINT | FK→partners(id) ON DELETE CASCADE | 배정 대상 |
| `assigned_by` | BIGINT | FK→users(id) ON DELETE SET NULL | 배정자 감사 |
| `status` | assignment_status | DEFAULT 'assigned' | |
| `created_at` | TIMESTAMP | DEFAULT now() | |
| `removed_at` | TIMESTAMP | NULL | |
| `deleted_at` | TIMESTAMPTZ | NULL | soft delete |
| `deleted_by` | BIGINT | FK→users(id) ON DELETE SET NULL | |

제약: UNIQUE `(record_id, partner_id)`
인덱스: `record_id`, `partner_id`, `idx_assignments_active (record_id, partner_id) WHERE deleted_at IS NULL`

> ⭐ 이 테이블이 "배정된 레코드만 접근" RBAC의 핵심입니다. `manager`/`member` 접근 시 서비스 레이어에서 `record_assignments` 조인으로 소유권을 강제합니다.

## 4. FK 관계도

```
users (id)
  ├─< partners.user_id            (1:1, CASCADE)
  ├─< records.deleted_by / restored_by      (SET NULL)
  ├─< record_assignments.assigned_by / deleted_by  (SET NULL)
  ├─< record_status_history.user_id (SET NULL)
  └─< record_memos.user_id        (CASCADE)

partners (id)
  ├─< record_assignments.partner_id (CASCADE)
  └─< record_memos.partner_id

records (id)               ← 핵심 중심 테이블
  ├─< record_assignments.record_id      (CASCADE)
  ├─< record_status_history.record_id   (CASCADE)
  ├─< record_memos.record_id            (CASCADE)
  ├─< identity_verifications.record_id  (SET NULL)
  └─> records.identity_verification_id  (SET NULL)

record_assignments (id)
identity_verifications (id)
verification_codes (id)    ← 독립 (FK 없음)
```

## 5. 마이그레이션 작성 가이드

원본처럼 `YYYYMMDDHHMMSS_<설명>.js` 파일을 `migrations/`에 순서대로 둡니다.
**의존성 순서**: ENUM → users → identity_verifications → records → partners → record_assignments → 나머지.

### 예시: 첫 스키마 마이그레이션

```javascript
// migrations/20260617000000_create_core_schema.js
exports.up = async (knex) => {
  // 1) ENUM
  await knex.raw(`CREATE TYPE role AS ENUM ('superadmin','admin','manager','member')`);
  await knex.raw(`CREATE TYPE account_status AS ENUM ('active','suspended')`);
  await knex.raw(`CREATE TYPE record_status AS ENUM ('new','assigned','in_progress','completed','cancelled')`);
  await knex.raw(`CREATE TYPE assignment_status AS ENUM ('assigned','removed')`);

  // 2) users
  await knex.schema.createTable('users', (t) => {
    t.bigIncrements('id').primary();
    t.string('username').notNullable().unique();
    t.string('password_hash').notNullable();
    t.specificType('role', 'role').notNullable();
    t.specificType('status', 'account_status').defaultTo('active');
    t.integer('failed_login_attempts').defaultTo(0);
    t.timestamp('locked_until').nullable();
    t.timestamp('password_changed_at').nullable();
    t.timestamp('suspended_at').nullable();
    t.timestamp('last_login_at').nullable();
    t.timestamp('created_at').defaultTo(knex.fn.now());
    t.timestamp('updated_at').defaultTo(knex.fn.now());
  });

  // 3) identity_verifications, records, partners, record_assignments ... (위 명세대로)
};

exports.down = async (knex) => {
  await knex.schema.dropTableIfExists('record_assignments');
  // ... 역순 drop
  await knex.raw('DROP TYPE IF EXISTS assignment_status');
  await knex.raw('DROP TYPE IF EXISTS record_status');
  await knex.raw('DROP TYPE IF EXISTS account_status');
  await knex.raw('DROP TYPE IF EXISTS role');
};
```

### 시드: superadmin 계정

```javascript
// seeds/20260617000001_superadmin.js
const bcrypt = require('bcrypt');

exports.seed = async (knex) => {
  const exists = await knex('users').where({ username: 'superadmin' }).first();
  if (exists) return;
  await knex('users').insert({
    username: 'superadmin',
    password_hash: await bcrypt.hash('superadmin123!', 10),
    role: 'superadmin',
    status: 'active',
    password_changed_at: null, // 첫 로그인 시 강제 변경
  });
};
```
